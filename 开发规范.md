# 开发规范文档

## 0. 优先级说明（重要）

- 本规范为**通用教学/作业级开发规范**，用于约束模板使用方式和一些通用约定。
- **每个具体任务/项目对应的《开发文档.md》（由扣子请求生成或下发）是该项目的核心需求文档。**
- 当 **《开发文档.md》 与本《开发规范.md》在「技术栈选型、功能范围、界面/UI 要求」等方面发生冲突时**：
  - **必须以对应项目目录下的《开发文档.md》为最高优先级**；
  - 本规范中的约定仅在**不与项目开发文档冲突**的前提下作为补充与参考；
  - 模板使用规则（如必须使用某个模板）也需要在**不破坏项目开发文档核心技术栈与功能目标**的前提下灵活适配。
- 简单理解：**先满足项目开发文档的业务和技术要求，再在不冲突的范围内尽量遵守本开发规范和模板约束。**

## 一、项目模板使用规范

### 1.1 模板目录说明

项目根目录下的 `project_template/` 文件夹包含以下可复用的项目模板框架：

- **express-backend/** - Express.js 后端框架模板
- **springboot-backend/** - Spring Boot 后端框架模板
- **Shopping-mall-template/** - 商城后端模板（TypeScript + Express）
- **html-frontend/** - HTML 前端模板
- **uniapp-stencil/** - UniApp 小程序模板
- **App-framework/** - Android 应用框架模板

### 1.2 模板使用原则

#### 原则一：优先使用模板
当项目需求匹配以下技术栈时，**必须**使用对应的模板框架：

1. **后端技术选型**
   - 如果技术选型包含 `Node.js`、`Express` → 使用 `express-backend` 模板`
   - 如果技术选型包含 `Java`、`Spring Boot`、`SpringBoot` → 使用 `springboot-backend` 模板
   - 如果技术选型包含 `商城`、`电商`、`购物`、`mall`、`ecommerce`、`TypeScript` + `Express` → 使用 `Shopping-mall-template` 模板

2. **前端技术选型**
   - 如果技术选型包含 `HTML`、`原生HTML`、`传统前端` → 使用 `html-frontend` 模板
   - 如果技术选型包含 `UniApp`、`uniapp`、`小程序` → 使用 `uniapp-stencil` 模板

3. **移动端技术选型**
   - 如果技术选型包含 `Android`、`Kotlin`、`移动应用` → 使用 `App-framework` 模板

#### 原则二：模板复制规范

1. **复制位置**
   - 模板应复制到 `user_project/{user_id}/{task_id}/` 目录下（task_id 用于区分同一用户的多个项目）
   - 根据项目结构，可以复制到根目录或创建子目录（如 `backend/`、`frontend/`）

2. **复制方式**
   - 使用 `shutil.copytree()` 或类似方法完整复制模板目录
   - **必须保留模板的完整目录结构**
   - **不要修改模板的核心配置文件**（如 `package.json`、`pom.xml` 等的基础配置）

3. **命名规范**
   - 如果项目包含多个模块，建议使用以下目录结构：
     ```
     user_project/{user_id}/{task_id}/
     ├── backend/          # 后端代码（从 express-backend 或 springboot-backend 复制）
     ├── frontend/         # 前端代码（从 html-frontend 或 uniapp-stencil 复制）
     └── 开发文档.md
     ```
   - **重要**：每个项目都有唯一的 `task_id`，用于区分同一用户的多个项目
   - 压缩包文件名使用 `{task_id}.zip`，而不是 `{user_id}.zip`

### 1.3 模板定制规范

#### 允许的修改
- ✅ 修改业务逻辑代码（在模板基础上开发）
- ✅ 添加新的路由、控制器、服务
- ✅ 修改数据库配置（连接信息）
- ✅ 添加项目特定的依赖包
- ✅ 修改应用名称、描述等元信息

#### 禁止的修改
- ❌ 删除模板的核心框架文件
- ❌ 修改模板的基础架构（如中间件、配置类）
- ❌ 删除模板的通用工具函数
- ❌ 修改模板的日志、错误处理机制

### 1.4 模板功能说明

- **配置文件**：`src/config/mysql.js`、`src/config/redis.js`

#### Android App Framework 模板
- **包含功能**：Kotlin、Gradle 构建、基础 Activity

- **技术栈**：TypeScript、Express、MySQL、Redis、JWT、Multer（文件上传）、Joi（参数验证）

- **工具类**：`utils/logger.ts`（日志工具）、`utils/response.ts`（响应处理）
- **使用方式**：在 `routes/` 目录下添加业务路由，在 `controllers/` 目录下创建控制器，在 `services/` 目录下实现业务逻辑
- **特色功能**：订单超时自动取消、支付状态检查、Redis 队列处理、定时任务调度

### 1.5 可选功能模块启用规范（强制规则）

模板可能包含“示例/增强”模块（如：JWT 认证、上传、Redis、MySQL、健康检查、前端后端适配层）。这些模块**必须遵循按需启用**原则：

1. **不强行使用**
   - 项目需求未要求时：允许保留代码文件，但**不得在入口处强制引用/挂载/执行**。

2. **入口处必须标注“可选”**
   - 后端：在 `src/app.js` 中，该模块所在代码块上方写一句：`// 可选：xxx（需要才启用）`。
   - HTML 前端：在 `index.html` 中用 HTML 注释标注可选 script（例如：`<!-- 可选：对接后端 -->`）。
   - UniApp：在页面或 README 中注明“示例页面默认不注册，需要手动加到 pages.json”。

3. **启用就启用全套依赖（不要半开半关）**
   - **上传**：启用 `/api/upload` 后，通常也要启用 `/uploads` 静态访问，否则拿到的 `url` 无法访问。
   - **JWT**：启用认证接口后，前端请求必须带 `Authorization: Bearer <token>`；token 建议统一用 `token` 存储键。
   - **MySQL/Redis**：启用连接前确保本地服务已启动、配置已按模板写死的默认值对齐。

4. **按需启用只允许用“入口是否引用/挂载/注册”控制**
   - 不允许为了“可选”引入复杂开关系统；保持模板简单可读。

## 二、通用开发规则


#### 状态码规范
- `200` - 成功
- `400` - 请求参数错误
- `401` - 未授权
- `403` - 禁止访问
- `404` - 资源不存在
- `500` - 服务器内部错误

### 2.4 错误处理规范

#### 异常处理
- **必须捕获所有异常**，不能抛出未处理的异常
- **记录错误日志**，包含错误堆栈信息
- **返回友好的错误信息**给前端，不暴露系统内部错误

#### 日志规范
- 使用模板提供的日志工具（如 `logger.js`、`Logger`）
- 日志级别：
  - `DEBUG` - 调试信息
  - `INFO` - 一般信息
  - `WARN` - 警告信息
  - `ERROR` - 错误信息

## 三、项目开发流程

### 3.1 项目初始化

1. **创建项目目录**
   - 在 `user_project/{user_id}/{task_id}/` 下创建项目目录（task_id 用于区分同一用户的多个项目）

2. **复制模板**
   - 根据技术选型，从 `project_template/` 复制对应模板
   - 如果项目包含前后端，分别复制对应模板到 `backend/` 和 `frontend/`

3. **配置环境**
   - 配置数据库连接信息
   - 配置 Redis 连接信息（如需要）
   - 配置 API 地址、端口等



### 3.2 开发阶段

1. **阅读开发文档**
   - 仔细阅读 `开发文档.md`，理解项目需求
   - 按照开发文档的模块划分进行开发

2. **遵循模板规范**
   - 在模板框架基础上开发，不要重新搭建框架
   - 使用模板提供的工具类和中间件

3. **完整实现开发文档中的全部模块（禁止假设与偷工减料）**
   - 必须严格按照 `开发文档.md` 中列出的**所有模块、页面、接口和功能点**逐项完整开发，不允许只实现一部分或只挑“核心功能”完成
   - 不允许以“时间不够 / 实现太复杂 / 看不太懂需求”等任何理由删减、弱化或跳过开发文档中已有的模块和功能
   - 不允许写到一半就停下，反向询问“剩下的是否可以不做 / 是否可以简化”，而是必须一次性把文档中约定的功能**全部写完、全部能跑通**
   - 严禁以“先假设一下”“先写个简单版本再说”“先占个位以后再补”为由写半成品代码，禁止只写接口壳子、空方法、空页面、仅含 TODO 注释或明显无法实际使用的“占位实现”
   - 当开发文档的描述不够细时，不允许以主观“假设需求”替代完整实现，更不允许因此少做或不做；应在现有描述的基础上，按照**功能完备、边界条件合理、可以真实使用**的标准，硬着头皮把该模块写完整



## 四、常见问题处理

### 4.1 模板相关问题

**Q: 如何判断应该使用哪个模板？**
A: 根据项目的技术选型字段（`task_technology`）判断：
- 包含 `Node.js`、`Express` → `express-backend`
- 包含 `Java`、`Spring Boot` → `springboot-backend`
- 包含 `商城`、`电商`、`购物`、`mall`、`ecommerce`、`TypeScript` + `Express` → `Shopping-mall-template`
- 包含 `HTML`、`原生` → `html-frontend`
- 包含 `UniApp`、`小程序` → `uniapp-stencil`
- 包含 `Android`、`Kotlin` → `App-framework`

**Q: 项目需要多个模板怎么办？**
A: 创建子目录分别复制：
```
user_project/{user_id}/{task_id}/
├── backend/     # 从 express-backend 或 springboot-backend 复制
├── frontend/    # 从 html-frontend 或 uniapp-stencil 复制
└── 开发文档.md
```

**Q: 可以修改模板的核心文件吗？**
A: 可以修改业务相关配置（如数据库连接），但不要删除或修改核心框架文件（如中间件、工具类）。

### 4.2 开发相关问题

**Q: 如何添加新的 API 接口？**
A: 
- Express：在 `src/routes/` 下创建路由文件，在 `app.js` 中注册
- Spring Boot：创建 Controller 类，使用 `@RestController` 和 `@RequestMapping`
- 商城模板（TypeScript）：在 `routes/` 下创建路由文件，在 `routes/index.ts` 中注册；在 `controllers/` 下创建对应的控制器

**Q: 如何处理数据库事务？**
A: 
- Express：使用模板提供的 `transaction` 方法
- Spring Boot：使用 `@Transactional` 注解


**Q: 如何使用缓存？**
A: 
- Express：使用 `src/config/redis.js` 提供的 Redis 客户端
- Spring Boot：使用 `RedisTemplate` 或 `@Cacheable` 注解

**Q: 遇到 "xxx is not a function" 错误怎么办？**
A: 这通常是**模块导入/导出不匹配**的问题。检查以下几点：
- **导出方式**：如果模块使用 `module.exports = { function1, function2 }` 导出对象，导入时必须使用**解构导入**
- **错误示例**：
  ```js
  // ❌ 错误：导出的是对象，但按函数导入
  const connectMySQL = require('./config/mysql');
  connectMySQL(); // TypeError: connectMySQL is not a function
  ```
- **正确示例**：
  ```js
  // ✅ 正确：使用解构导入
  const { connectMySQL } = require('./config/mysql');
  connectMySQL(); // 正常调用
  ```
- **判断方法**：查看模块文件末尾的导出语句：
  - `module.exports = functionName` → 直接导入：`const func = require('./module')`
  - `module.exports = { func1, func2 }` → 解构导入：`const { func1 } = require('./module')`
  - `exports.func1 = ...` → 解构导入：`const { func1 } = require('./module')`

**Q: 遇到 express-rate-limit 的 "ERR_ERL_UNEXPECTED_X_FORWARDED_FOR" 或 "ERR_ERL_PERMISSIVE_TRUST_PROXY" 错误怎么办？**
A: 这是 `express-rate-limit` 与 Express `trust proxy` 配置相关的问题。解决方法：
- **问题1：ERR_ERL_UNEXPECTED_X_FORWARDED_FOR**
  - 原因：检测到了 `X-Forwarded-For` 请求头，但 Express 的 `trust proxy` 设置未启用
  - 解决：添加 `app.set('trust proxy', 1)` （只信任第一层代理，更安全）
  
- **问题2：ERR_ERL_PERMISSIVE_TRUST_PROXY**
  - 原因：`trust proxy` 设置为 `true` 太宽松，可能被绕过限流
  - 解决：
    ```js
    // ✅ 推荐配置（开发环境）
    app.set('trust proxy', 1); // 只信任第一层代理
    const limiter = rateLimit({
      // ... 其他配置
      validate: false // 禁用严格验证，适合开发环境
    });
    ```
- **原因**：当应用部署在反向代理（如 Nginx）后面或使用开发代理时，需要信任代理才能正确获取客户端真实IP
- **注意**：生产环境建议使用 `app.set('trust proxy', 1)` 并移除 `validate: false`，以获得更好的安全性

**创建数据库时候注意**
禁用外键检查（避免创建表时的外键约束问题）
修复 SQL 语法错误：外键约束中 ON SET NULL 应为 ON DELETE SET NULL。

**Q: 前端遇到 CORS 错误，提示 "Request header field xxx is not allowed by Access-Control-Allow-Headers" 怎么办？**
A: 这是跨域资源共享（CORS）配置问题，通常发生在以下情况：
- **问题原因**：
  - 前端（如 `http://localhost:8080`）访问后端（如 `http://localhost:3000`）属于跨域请求
  - 前端在请求中使用了自定义请求头（如 `user-id`、`token` 等）
  - 浏览器在发送带有自定义请求头的跨域请求前，会先发送 OPTIONS 预检请求
  - 如果后端 CORS 配置的 `allowedHeaders` 中没有包含该自定义请求头，浏览器会拒绝请求

**解决方法**：
1. **在 Express 后端配置中添加自定义请求头**（在 `src/app.js` 中）：
   ```js
   app.use(cors({
     origin: ['http://localhost:3000', 'http://localhost:8080'],
     credentials: true,
     methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH', 'OPTIONS'],
     allowedHeaders: ['Content-Type', 'Authorization', 'X-Requested-With', 'user-id', 'token'] // 添加自定义请求头
   }));
   ```


3. **注意事项**：
   - 所有前端使用的自定义请求头都必须在后端的 `allowedHeaders` 中声明

**Q: 前端遇到 "NavigationDuplicated" 错误和 401 Unauthorized 怎么办？**
A: 这是 Vue Router 的常见问题，通常发生在以下情况：
- **NavigationDuplicated**：尝试导航到当前已经在的页面
- **401 Unauthorized**：Token 过期或无效，API 拦截器尝试跳转到登录页

**解决方法**：
1. **在 API 拦截器中避免重复导航**：
   ```js
   if (error.response.status === 401) {
     if (router.currentRoute.path !== '/login') {
       router.replace('/login').catch(err => {
         if (err.name !== 'NavigationDuplicated') {
           console.error(err)
         }
       })
     }
   }
   ```

2. **修复 Vue Router 重复导航警告**（在 router/index.js 中）：
   ```js
   // 修复 push 方法
   const originalPush = VueRouter.prototype.push
   VueRouter.prototype.push = function push(location) {
     return originalPush.call(this, location).catch(err => {
       if (err.name !== 'NavigationDuplicated') {
         return Promise.reject(err)
       }
     })
   }
   
   // 修复 replace 方法
   const originalReplace = VueRouter.prototype.replace
   VueRouter.prototype.replace = function replace(location) {
     return originalReplace.call(this, location).catch(err => {
       if (err.name !== 'NavigationDuplicated') {
         return Promise.reject(err)
       }
     })
   }
   ```

3. **在路由守卫中检查当前路径**：
   ```js
   if (!store.state.isLogin) {
     if (to.path !== '/login') {
       next('/login')
     } else {
       next()
     }
     return
   }
   ```

**Q: 后端遇到 "Cannot read properties of undefined (reading 'total')" 错误怎么办？**
A: 这通常发生在分页查询中，当 `countResult[0]` 为 `undefined` 时。常见原因：
- SQL 查询返回空结果
- SQL 替换逻辑有问题（如使用 `sql.replace(/SELECT.*FROM/, ...)` 时正则匹配不正确）

**解决方法**：
1. **添加安全检查**：
   ```js
   const countResult = await query(countSql, params);
   // ❌ 错误：直接访问可能为 undefined
   const total = countResult[0].total;
   
   // ✅ 正确：添加安全检查
   const total = (countResult && countResult[0] && countResult[0].total) 
     ? countResult[0].total 
     : 0;
   ```

2. **更好的方法：构建独立的 COUNT 查询**：
   ```js
   // 构建 WHERE 条件
   let whereClause = 'WHERE 1=1';
   const params = [];
   // ... 添加条件 ...
   
   // COUNT 查询
   const countSql = `SELECT COUNT(*) as total FROM table ${whereClause}`;
   const countResult = await query(countSql, params);
   const total = (countResult && countResult[0] && countResult[0].total) 
     ? countResult[0].total 
     : 0;
   
   // 数据查询
   const dataSql = `SELECT * FROM table ${whereClause} ORDER BY ... LIMIT ...`;
   ```

3. **避免使用正则替换 SQL**：`sql.replace(/SELECT.*FROM/, ...)` 在多行 SQL 中可能匹配不正确，建议分别构建 COUNT 和 SELECT 查询。

**Q: 登录接口返回 401 Unauthorized 或密码验证失败怎么办？**
A: 这通常是因为数据库中的密码不是真正的 bcrypt 加密值。解决方法：
1. **生成真正的 bcrypt 加密值**：
   ```bash
   # 在 Node.js 中
   node -e "const bcrypt = require('bcrypt'); bcrypt.hash('your_password', 10).then(hash => console.log(hash));"
   ```
2. **更新数据库中的密码**：
   ```sql
   UPDATE sys_user SET password = '$2b$10$...' WHERE username = 'admin';
   ```
3. **或者在代码中重置密码**：创建一个临时路由来重置密码
4. **检查密码验证逻辑**：确保使用 `bcrypt.compare()` 而不是直接比较字符串

**Q: 前端请求返回 500 错误，如何调试？**
A: 后端 500 错误通常表示服务器内部错误，调试方法：
1. **查看后端日志**：检查控制台输出的错误堆栈信息
2. **添加详细错误日志**：
   ```js
   catch (error) {
     logger.error('Error message:', error.message);
     logger.error('Error stack:', error.stack);
     res.error(error.message || '操作失败', 500);
   }
   ```
3. **检查数据库连接**：确保数据库已启动且表已创建
4. **检查 SQL 语法**：确保 SQL 语句正确，参数数量匹配
5. **检查数据格式**：确保传入的数据格式正确

### 5.1 学生作业级别简化约定（重点，减少常见 Bug）

为了降低同学们在开发时踩坑的概率，这里做一些**有意的简化约定**，只要求能跑通、功能正确，不追求企业级完备性：

1. **SQL 中 LIMIT / OFFSET 的写法（强制要求）**
   - 分页相关的 `LIMIT` / `OFFSET` 不要再使用占位符：  
     **禁止：**
     ```js
     // ❌ 容易因为类型问题或驱动差异出错
     const sql = 'SELECT * FROM work_experience LIMIT ? OFFSET ?';
     const rows = await query(sql, [limitNum, offset]);
     ```
     **推荐：**
     ```js
     // ✅ 先把数字处理干净，再拼接到 SQL 里
     const safeLimit = parseInt(limitNum, 10) || 10;
     const safeOffset = parseInt(offset, 10) || 0;
     const sql = `
       SELECT * FROM work_experience
       ORDER BY sort DESC, start_date DESC
       LIMIT ${safeLimit} OFFSET ${safeOffset}
     `;
     const rows = await query(sql);
     ```
   - 关键点：**一定先用 `parseInt` 把页码、条数转成数字并做上下限控制**，再拼接到 SQL，避免类型不对导致 SQL 报错。

2. **数据库 / Redis 简化配置（本地作业环境）**
  - 为了减少「连不上库」这类问题，本项目统一推荐**本地作业环境**使用下面的默认配置（只用于课程作业，禁止用于线上环境）：
    - MySQL：
      - `host = localhost`
      - `port = 3306`
      - `user = root`
      - `password = root`
      - `database = personal_cms`
    - Redis：
      - `host = localhost`
      - `port = 6379`
      - `password = root`（如果本地没设密码，可以留空，同时把代码里的密码也留空）
  - **本规范要求：不要使用 `.env` 等环境变量文件；直接在配置文件中写死上述默认值**（仅限本课程本地作业环境，真实生产环境请根据实际需求使用安全的配置管理方案）。同学只需要**保证本地 MySQL / Redis 按这个配置启动即可**，避免因为环境差异导致连接失败。

3. **JWT / 安全相关的简化**
   - 学生作业可以使用固定的 `JWT_SECRET`（例如：`JWT_SECRET=student-demo-secret`），不要求做复杂的密钥管理。
   - 只要能完成「登录成功 → 能访问后台接口；未登录/过期 → 不能访问后台接口」这条主线即可。

4. **Redis 可选**
   - 如果同学本机没有 Redis 或装 Redis 比较困难，可以：
     - 保留 Redis 相关代码，但在连接失败时只打日志，不影响项目启动；
     - 或者在作业中**不启用缓存逻辑**，直接从 MySQL 查数据即可。

5. **优先保证「能跑通 + 功能对」**
   - 对于学生作业，更重要的是：
     - 能启动项目（前后端都能跑起来）；
     - 主要功能（登录、增删改查）按开发文档要求能走通；
   - 数据库sql文件可以适当写一些假数据。
