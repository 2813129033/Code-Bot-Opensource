## 🧠 一、项目整体思路（像工程师一样工作的 AI）

从工程视角看，这个项目把“AI 写代码”强行拆成 **几个非常清晰的阶段**：

### 1. 输入层：只有一个“真需求”来源

- 你要么提供一份**开发文档**（`开发文档.md`）
- 要么提供一段**自然语言需求**（比如：“做一个在线考试系统……”）
- 系统会调用规划智能体接口，**自动生成统一格式的开发文档**

> 重点：**所有后续步骤，都必须围绕这份开发文档展开**，  
> 不允许“AI 自己想一套”“写着写着改需求”。

### 2. 任务层：把需求拆成可交付的任务树

- 有了 `开发文档.md` 之后，系统会让 AI 根据文档生成一个 `task_tree.json`：
  - 最外层是多个 **模块（module）**
  - 每个模块下面是多个 **任务（task）**
  - 每个任务有：`name`、`description`、`type`、`status` 等字段
- 这就是项目的 **单一事实源（SSOT）**：
  - **所有开发工作，只能围绕这个任务树推进**
  - 每个任务都有清晰的责任边界和完成状态

> 可以理解为：  
> `task_tree.json = 机器可读的“项目排期 + 任务清单”`

### 3. 执行层：AI 逐任务推进，而不是“一把梭”

`python-auto` 下的脚本负责整个执行流程，大体做三件事：

1. **找到下一个未完成任务**
   - 通过 `task_checker.py` 里的 `find_next_unfinished_task(...)`
   - 只看状态为 `pending / in_progress / failed` 的任务
2. **让 AI 专注完成这个任务**
   - 生成一个**针对单个任务的提示词**（`generate_task_check_prompt(...)`）
   - 引导 AI：检查代码 → 实现 / 修复代码 → 更新 `task_tree.json` 中对应任务的 `status`
3. **记录结果并循环**
   - 如果任务完成 → 标记为 `completed`
   - 如果任务失败 / 有问题 → 标记为 `failed` 并写入原因
   - 然后继续找下一个任务，直到所有任务 `completed`

> 关键点：**AI 不能随便“顺手改别的”**，  
> 只能围绕“当前任务”修改代码并更新任务状态。

### 4. 收尾层：结构完整度的最终检查

即使所有任务都标记为 `completed`，系统还会做一次 **“Final Check”**：

- 通过 `generate_final_check_prompt(...)` 生成**最终检查提示词**
- 引导 AI：
  - 把 `task_tree.json` 和实际代码一一对照
  - 找出“看似完成但其实没写完 / 写不全 / 只是空壳”的地方
  - 对应任务重新标记为 `pending / failed`，并在 `final_check_notes` 写明原因
  - 维护顶层字段 `final_check_count`（检查过几轮）

> 只有在：
> - 所有任务 `status = completed`
> - `final_check_count >= 1`  
> 的情况下，系统才认为：**这个项目可以对外“交付”**。

## 🔁 三、完整使用流程（从 0 到 1）

下面以“在本机跑一个完整流程”为例，做一个**通俗版流程说明**。  
为了便于理解，我们抽象成六步：

### 第 0 步：准备项目根目录

项目根目录里已包含：

- `python-auto/`：自动执行引擎（任务调度、开发文档生成、任务树推进等）
- `project_template/`：可复用的项目模板（Express 后端、Spring Boot、HTML 前端、UniApp、小程序等）
- `开发规范.md`：统一的开发规范说明
- `项目描述.md`：项目背景 / 设计理念说明（你已经看过）

你要做的只是一件事：**不要随便改动这些目录结构**。

### 第 1 步：创建用户项目目录

系统默认使用：

- `user_project/{user_id}/{task_id}/` 作为每个项目的工作目录

其中：

- `user_id`：可以是 QQ 号、学号、员工号等
- `task_id`：同一用户下，用来区分不同项目

一个典型的工程目录会长这样：

```text
user_project/
  └── 123456/                # user_id
      └── 20250301-001/      # task_id
          ├── 开发文档.md
          ├── backend/       # 可选：后端模板代码
          └── frontend/      # 可选：前端模板代码
```

> 小结：**所有与某个具体项目有关的文件，都只放在它自己的 `{user_id}/{task_id}` 目录下**。

### 第 2 步：生成/准备《开发文档.md》

有两种方式：

- **方式 A：你自己写好《开发文档.md》**
  - 直接把它放到 `user_project/{user_id}/{task_id}/开发文档.md`
  - 内容越清晰越好，至少要写清：模块、接口、页面、数据结构等

- **方式 B：让系统帮你生成《开发文档.md》**
  - 系统会调用远程规划接口（在 `python-auto/auto.py` 中配置）
  - 根据你提供的自然语言需求，自动生成 `开发文档.md` 并保存

无论哪种方式，**后面的所有工作都以这份文档为准**。

### 第 3 步：生成 `task_tree.json`（任务树）

在有了 `开发文档.md` 后，系统会：

1. 调用 AI，对文档进行模块化拆解
2. 生成一个结构大致如下的 `task_tree.json`：

```json
{
  "modules": [
    {
      "name": "用户模块",
      "tasks": [
        {
          "name": "用户注册接口",
          "description": "...",
          "type": "api",
          "status": "pending"
        }
      ]
    }
  ]
}
```

> 你可以理解为：**把“自然语言需求”变成“机器可读的工作清单”**。

### 第 4 步：AI 逐任务开发 & 标记状态

核心逻辑由 `python-auto` 中的脚本负责，大致流程是：

1. 读取 `task_tree.json`
2. 用 `find_next_unfinished_task(...)` 找出下一个需要处理的任务
3. 根据任务信息生成提示词，引导 AI：
   - 检查当前代码是否已经实现了这个任务
   - 如果没有实现 / 不完整 → 写代码 / 补代码
   - 完成后修改 `task_tree.json` 中对应任务的 `status` 为 `"completed"`
4. 重复上述步骤，直到：
   - 所有任务都变成 `"completed"`，但此时还没正式“交付”

在这个过程中，系统还会：

- 记录每个任务的失败原因
- 限制重试次数，避免无限死循环
- 定期清理资源，避免长期运行卡死

> 你可以把这一阶段理解为：  
> **“AI 开发工程师在一条条勾选自己的任务清单”**。

### 第 5 步：最终结构检查（Final Check）

当所有任务都标记为 `"completed"` 后，还要再走一遍 **Final Check**：

- 系统会调用 AI 执行“结构完整度检查”：
  - 对照 `task_tree.json` 和代码实现
  - 把“看起来完成但实际上没实现好的任务”重新标记为 `pending / failed`
  - 在 `final_check_notes` 写清楚“具体少了什么 / 哪个接口不完整”
  - 增加顶层 `final_check_count` 次数

只有在：

- `all_tasks_completed(task_tree) == True`
- 且 `final_check_count >= 1` 时

系统才认为：**整个项目通过了“交付门槛”**。

### 第 6 步：打包 / 交付 / 归档

当一个项目最终通过校验后，可以：

- 把 `user_project/{user_id}/{task_id}/` 目录打成 zip 包，放到：
  - `user_project_zip/{user_id}/{task_id}.zip`
- 或通过你们对接的渠道（QQ 机器人 / Web）自动发送给用户

> 这一步的实现方式比较灵活，  
> 本仓库提供的是**底层目录规范与逻辑支撑**。

## 🖱️ 四、GUI 自动化是怎么“点按钮”的？（图片识别原理 + 可自定义）

很多人一开始会有一个疑问：

> “你既然说这是一个自动化工作流，那它到底是怎么在 Coze / 浏览器 / 客户端里点按钮、输提示词的？”

答案是：**这一层完全是靠「截图 + 图片识别 + 鼠标键盘模拟」完成的。**

### 1. 核心原理：靠截图找到按钮，再模拟人工点击

在 `python-auto/auto.py` 里，你会看到：

- 使用了 `pyautogui` 来做：
  - `locateCenterOnScreen(...)`：在当前屏幕上寻找一张小图片出现的位置
  - `moveTo(...)`、`click()`：把鼠标移动到那个位置，并点击
  - `hotkey("ctrl", "a")` / `hotkey("ctrl", "v")` / `press("delete")`：模拟键盘操作
- 配套有一个 `btnimg/` 目录，用来存放各种按钮的截图：
  - `input.png` / `input2.png`：LLM 输入框区域
  - `send.png`：发送按钮
  - `send-success.png`：发送成功后的状态标识
  - `newpro.png`：新建项目相关按钮
  - `Switchaccount.png` / `Confirmswitch.png` / `Switchingcomplete.png`：切换账号流程中用到的按钮

自动化流程的大致套路是：

1. **寻找输入框**
   - 先用 `wait_and_click("btnimg/input.png", ...)`，找不到再试 `input2.png`
   - 找到后点击，让光标聚焦到输入框
2. **清空并输入提示词**
   - `ctrl + a` 全选 → `backspace/delete` 清空
   - 利用剪贴板粘贴完整提示词（必要时逐字 `write`）
3. **点击发送按钮**
   - 用 `wait_and_click("btnimg/send.png", click_times=1)` 找到发送按钮并单击
4. **识别是否发送成功**
   - 等待一小段时间，在屏幕上寻找 `send-success.png`
   - 找到则认为“这次发送成功”，否则按配置重试

比如 `_generate_task_tree_with_ai(...)`、`send_continue_prompt(...)` 这些函数，内部就是按上面的“找输入框 → 输入内容 → 点发送 → 等待成功”的固定节奏来写的。

### 2. 为什么要这样设计（而不是绑死 Cursor 或某个 IDE）？

这么做有几个好处：

- **不依赖特定 IDE / 浏览器插件**
  - 你不需要一定用 Cursor，也不需要改 Coze 的内部接口
  - 只要屏幕上有一个可以输入提示词、点击发送、看到结果的界面，这套脚本就能“看图说话”地操作它
- **行为完全等价于“真人在用界面”**
  - 所有步骤都必须走过真实 GUI 流程
  - 如果按钮位置变了 / 主题改了 / 加载变慢了，你也会在界面上肉眼看到问题，更容易调试
- **方便非工程同学调整**
  - 很多修改只需要“重截一张图 + 换个文件名”，不需要写一行后端代码

代价是：

- 分辨率、缩放比例、主题、语言包都会影响识别效果；
- 所以仓库里同时准备了 `input.png` 和 `input2.png` 两张输入框截图，用来兼容不同 UI 状态。

### 3. 你可以怎样“自行修改 / 定制”这些 GUI 步骤？

如果你觉得：

- 自己的 Coze / 浏览器布局和作者当时不一样；
- 想接入另一个 Web IDE / 客户端；
- 或者只是想调整“先点哪里，再点哪里”的顺序，

那么你完全可以改这套 GUI 流程，主要有三种方式：

- **（1）替换按钮截图**
  1. 打开你要用的界面（浏览器里的 Coze / 其他 Web 工具 / 自己的后台等）
  2. 用系统截图工具截取关键区域的小图：
     - 输入框（只截中间文本区域，避免截太大）
     - 发送按钮 / 确认按钮
     - 发送成功 / 操作成功的提示区域（如果有固定图标）
  3. 将这些小图保存为：
     - `python-auto/btnimg/input.png`
     - `python-auto/btnimg/send.png`
     - `python-auto/btnimg/send-success.png`
     - 以及你需要用到的其他新文件（例如 `btnimg/newchat.png`）
  4. 如果你改了文件名，就到 `auto.py` 里对应的位置，把 `"btnimg/xxx.png"` 换成你自己的名字。

- **（2）调整识别参数（容错程度 & 耐心程度）**
  - 在 `auto.py` 顶部 `CONFIG` 里：
    - `min_confidence`：图片匹配最低置信度（0.6–0.9）  
      - 太高：可能识别不到按钮  
      - 太低：可能误把别的地方当按钮
    - `max_retry_attempts`：同一操作失败后重试几次
    - `project_wait_time` / `modify_project_wait_time`：等 AI 写代码的时间长短
  - 在 `wait_and_click(...)` 调用处还可以单独传：
    - `timeout`：最多等多少秒
    - `click_times`：点击次数（有的按钮只需要单击，有的可以双击）

- **（3）改“步骤顺序”和“在哪一步点哪个按钮”**
  - 所有 GUI 步骤都写在 `auto.py` 里，形式其实就是：
    - `wait_and_click("btnimg/xxx.png")`
    - `pyautogui.hotkey(...)`
    - `time.sleep(...)`
  - 你可以按自己的界面真实使用路径，重新排布：
    - 比如在 `_generate_task_tree_with_ai(...)` 前先点一次“新建对话”按钮：
      - 截一张 `btnimg/newchat.png`
      - 在函数开头加一行：`wait_and_click("btnimg/newchat.png", ...)`

> 换句话说：  
> **这套 GUI 步骤不是“写死的黑盒流程”，而是可以被你当做“半可视化脚本”随时调整。**

### 4. 进阶方向：截整屏交给 AI 决定“下一步点哪里”（欢迎大佬共建）

目前仓库里的实现，是**基于固定截图模板**的：  
> “我先截好一批按钮图，然后在代码里写死：第 A 步点这个、第 B 步点那个。”

如果有同学愿意进一步升级这套系统，其实可以做成更智能的形态：

- **1）运行时实时截图**
  - 周期性用 `pyautogui.screenshot()`（或其他库）截取当前窗口 / 全屏画面
  - 将图片压缩后发送给一个“界面理解类”模型（可以是多模态模型）
- **2）让 AI 决定“当前界面是什么状态 / 下一步该点哪里”**
  - 提示词里描述当前任务意图（比如“我要在 Coze 里新建一个对话并发送以下提示词”）
  - 让模型在截图上标出：
    - 下一步要点击区域的大致坐标 / 位置描述
    - 或者直接返回“按钮语义 + 坐标”的结构化信息
- **3）Python 侧根据 AI 返回的坐标执行点击**
  - 用 `pyautogui.moveTo(x, y)` + `click()` 完成实际操作
  - 如果模型判断“当前界面不在预期流程中”，可以直接让它返回错误信息并中断

这样一来：

- 就不再依赖特定的 `btnimg/*.png` 模板；
- UI 改版 / 按钮位置轻微变化时，AI 也有机会自动适应；
- 对有多模态 API / 本地视觉模型资源的大佬来说，这是一个非常适合 PR 的扩展方向。

目前仓库暂时没有内置这层“AI 看截图自己决定点哪里”的逻辑，  
**但整个流程设计已经为这种扩展预留了空间**：只要你能把“下一步点击行为”封装成一个函数，就可以无缝接入现有工作流（替换掉现在的 `wait_and_click(...)` 等调用即可）。

## 🧪 五、如何判断“这个项目算真正完成了”？

可以用一个非常简单的 checklist 来判断：

- **1. 文档层**
  - `user_project/{user_id}/{task_id}/开发文档.md` 存在且内容完整
- **2. 任务层**
  - 同目录下存在 `task_tree.json`
  - `modules` 中的所有任务 `status` 都为 `"completed"`
- **3. 检查层**
  - `task_tree.json` 顶层存在 `final_check_count`
  - `final_check_count >= 1`
  - 所有任务的 `final_check_notes` 要么不存在，要么为空字符串
- **4. 运行层**
  - 项目能正常启动（前端/后端/小程序各自能跑）
  - 开发文档里的核心功能流程都能真实走通

当以上四层都达成时，你可以安心地说一句：  
> **“这个项目已经达到可交付标准了。”**
